# 统一运维控制面 —— 任务拆分

> 基于《统一运维控制面-设计文档 v0.2》。三期全部细拆到任务级,颗粒度统一。
> 后续实现按任务 ID 引用。设计文档章节以 §N 形式交叉引用。

## 技术栈(定稿)

| 层 | 选型 | 理由 |
|----|------|------|
| 后端框架 | Python 3.11+ / FastAPI / Uvicorn | 异步、自带 OpenAPI、Pydantic 边界校验,契合 REST + WebSocket |
| ORM / 迁移 | SQLAlchemy 2.0 (async) + Alembic | 关系清晰、迁移可控 |
| 数据校验 | Pydantic v2 | 系统边界统一校验 |
| 任务队列 | Celery + Redis | 长任务、重试、beat 定时(轮询兜底),对应文档 Asynq 角色;Flower 可观测 |
| SSH 执行 | AsyncSSH(点对点)+ ansible-runner(批量幂等,可选) | Python 原生契合 |
| Agent 通信(二/三期) | gRPC (grpcio) 双向流 | 长连、服务端主动下发;Agent 本体建议 Go 单二进制分发(proto 共享),PyInstaller 备选 |
| 凭证保险箱 | HashiCorp Vault (hvac);MVP 先 Fernet + 外部 KMS 主密钥过渡 | 接口预留,不落明文 |
| 监控 | Prometheus + node_exporter + Alertmanager;Grafana iframe 内嵌 | 事实标准 |
| 前端 | React 18 + TS + Vite;TanStack Query + Zustand;Ant Design;ECharts | 服务端/本地状态分离;曲线自绘 + 内嵌 Grafana |
| 数据库 | PostgreSQL 15+ | 事务可靠 |
| 可观测 | structlog 结构化日志 + 请求追踪 | |

## 约定

- 任务 ID:`T<epic>.<seq>`(例 `T1.3`);依赖用任务 ID 标注。
- 每任务含:**依赖 / 产出 / 验收**。
- **Epic 0** 是三期共同地基;**Epic 4(Agent)** 非 MVP 关键路径,按需启动(§5.3)。
- 颗粒度目标:单任务约 0.5~3 人日,可独立开发与验收。
- 期次对应设计文档 §17:Epic 1 = 第一版(运行态),Epic 2 = 第二版(部署配置),Epic 3 = 第三版(交付态关联)。

## Epic 0 — 项目地基(三期共用,先行)

> 对应设计:§13 安全底座、§2 设计原则、§16 技术选型。此 Epic 不产出业务功能,但后续所有 Epic 依赖它。

### T0.1 后端工程骨架
- 依赖:无
- 产出:FastAPI 应用骨架、分层目录(`api/ core/ models/ services/ adapters/ workers/`)、配置加载(pydantic-settings,读环境变量)、健康检查 `/healthz`、结构化日志(structlog)、统一异常处理与 API 响应封装(§patterns 响应 envelope)。
- 验收:本地 `uvicorn` 起服务,`/healthz` 返回 200,日志为结构化 JSON。

### T0.2 数据库与迁移基线
- 依赖:T0.1
- 产出:PostgreSQL 连接(async SQLAlchemy)、Alembic 初始化、连接池配置、事务上下文管理。空迁移基线可 `upgrade/downgrade`。
- 验收:`alembic upgrade head` 成功,建库连通,健康检查含 DB 探活。

### T0.3 任务队列基础设施
- 依赖:T0.1
- 产出:Celery + Redis 接线、worker 启动、beat(定时)、Flower 面板;通用任务落库骨架对齐 `tasks` 表(§14.6),含 `pending/running/success/failed/unknown` 状态机。
- 验收:提交一个样例异步任务,worker 执行,`tasks` 表状态流转正确,Flower 可见。

### T0.4 认证与 RBAC 骨架
- 依赖:T0.2
- 产出:用户/角色/权限模型;登录发 JWT;依赖注入式权限校验(精确到"资源+环境+动作",§10.2/§13);种子管理员。**MVP 权限点可粗,但接口形态定死,后续细化不改调用方。**
- 验收:未授权 401、越权 403;prod 删除类动作要求授权角色。

### T0.5 凭证保险箱抽象层
- 依赖:T0.2
- 产出:`SecretStore` 接口(get/put/rotate);两实现——`VaultSecretStore`(hvac)与 `LocalSecretStore`(Fernet+KMS 主密钥,MVP 用);凭证按 id 引用,**严禁明文落业务表**(§13)。
- 验收:SSH 私钥/平台 token 存取走保险箱,业务表只存 `credential_id`;切换实现不改调用方。

### T0.6 审计日志基座
- 依赖:T0.2
- 产出:`audit_logs` 表(§14.7)+ 写审计的中间件/装饰器;所有写操作记录 actor/action/target/env/before/after/result/ip/ua;仅追加、不可改。
- 验收:任一写操作后审计表有对应不可篡改记录,可按资源/人/环境检索。

### T0.7 前端工程骨架
- 依赖:无
- 产出:React+TS+Vite 骨架;路由;Ant Design 主题;TanStack Query + Axios 封装(统一错误/鉴权头);登录页 + 受保护路由;布局壳(侧边导航:服务器/服务/部署/配置/监控)。
- 验收:登录后进入布局壳,401 自动跳登录,一个联调用的占位页能跑通请求。

### T0.8 本地开发环境编排
- 依赖:T0.1, T0.2, T0.3
- 产出:`docker-compose`(postgres/redis/后端/前端/prometheus/grafana/alertmanager)、Makefile 常用命令、`.env.example`、README 起步说明。
- 验收:`make up` 一键拉起全栈,前后端与依赖组件互通。

### T0.9 CI 与测试基线
- 依赖:T0.1, T0.7
- 产出:后端 pytest + 覆盖率门槛;前端 vitest + 组件测试;lint/format(ruff+black / eslint+prettier);CI 流水线(装依赖→lint→test→build)。
- 验收:CI 绿;覆盖率报告产出(目标 80%,§testing)。

### T0.10 实时推送通道(WebSocket)
- 依赖:T0.1, T0.4
- 产出:WebSocket 端点(带 JWT 鉴权握手)+ 连接管理器(按 topic/资源订阅);服务端在任务状态流转、部署 feed、告警到达时向订阅者推送(§2「UI 轮询/推送」、§3 REST/WebSocket)。前端 T1.17/T2.17 优先走推送,轮询作降级兜底。
- 验收:客户端订阅某 task_id,后端状态流转时收到推送;未鉴权连接被拒;断线后可重连并恢复订阅。

### T0.11 API 契约与前端类型生成
- 依赖:T0.1, T0.7
- 产出:后端导出 OpenAPI schema;前端由 OpenAPI 生成 TS 类型(openapi-typescript),纳入 CI 校验(schema 变更未同步则失败)。统一 API 响应 envelope 的前端解包工具。
- 验收:后端改动接口契约后,前端类型生成产物同步更新;类型不一致时 CI 报错。

### T0.12 API 网关中间件:限流与安全响应头
- 依赖:T0.1
- 产出:全局限流(按用户/IP,令牌桶,Redis 计数)、安全响应头(CORS 白名单、`X-Content-Type-Options`、`X-Frame-Options` 等)、请求体大小限制;可按路由豁免(webhook 走 §8.3 自身鉴权)。
- 验收:超限返回 429;安全响应头存在;webhook 路由不被误限流。

## Epic 2 — 部署与配置(设计文档「第二版」)

> 目标:能触发部署、被动感知部署回流、部署 feed、配置版本化与一键回滚、多环境与 prod 审批。对应 §8、§9、§10、§12、§14.3/14.5、§15.2/15.3。

### 部署记录与回流

#### T2.1 deployments 表与迁移
- 依赖:T0.2, T1.2
- 产出:`deployments` 表(§14.3,含 `previous_deployment_id/source/scan_result_id/strategy/pipeline_*`)+ 状态机(`running→success/failed/rolled_back`,只前进不回退)+ repository。
- 验收:迁移与单测通过;状态机拒绝非法回退。

#### T2.2 CI 适配层接口 + Jenkins/GitLab 实现
- 依赖:T0.5
- 产出:`PipelineAdapter` 接口(`trigger/get_status/get_logs`,§8.1);Jenkins(buildWithParameters)与 GitLab(trigger token)两实现;平台 token 走保险箱。

- 依赖:T2.1, T2.2, T0.3, T0.4, T0.6

#### T2.4 Webhook 安全基座:HMAC 验签 + 防重放(§8.3)

#### T2.5 幂等与乱序保护基座(§8.3)

#### T2.6 部署回流 Webhook(模式 B)

#### T2.7 轮询兜底(补偿通道)
- 依赖:T2.2, T2.6

## 依赖顺序与里程碑

**关键路径(建议顺序):**

```
Epic 0(地基,尽量并行)
  └─▶ Epic 1(运行态)  ── 里程碑 M1:能看能操作能看资源
        └─▶ Epic 2(部署与配置) ── 里程碑 M2:部署回流 + 配置回滚 + prod 审批
              └─▶ Epic 3(交付态关联) ── 里程碑 M3:全链路关联 + 发布策略 + 告警联动
Epic 4(Agent)── 与上述解耦,规模到位后并行插入,替换 T1.5/T1.12 占位
```

**并行建议:**
- Epic 0 内:后端线(T0.1→0.6)、前端线(T0.7)、环境线(T0.8/0.9)可三路并行。
- 每个 Epic 内:数据模型任务先行,执行器/adapter 与前端页面可在 API 定稿后并行。
- 前端页面任务依赖对应后端 API 完成契约即可开工(可先按契约 mock)。

**跨 Epic 复用要点:**
- §8.3 Webhook 安全+幂等基座(T2.4/T2.5)被 T2.6、T3.2、T3.10 复用。
- 生命周期异步执行(T1.10)被部署/回滚(T2.3/T2.8)复用。
- 保险箱(T0.5)贯穿 SSH 私钥、CI token、扫描/告警 secret、配置敏感值。
- 实时推送通道(T0.10)被 T1.17(任务进度)、T2.17(部署 feed)、T3.14(告警)复用;轮询(T1.11)为降级兜底。
- 限流与安全头(T0.12)、API 类型生成(T0.11)为全局横切,所有 Epic 的接口默认适用。
