## 1. 环境准备

### 1.1 版本要求

| 组件 | 版本要求 | 说明 |
|------|----------|------|
| Python | 3.11+ | 后端运行时 |
| uv | 最新 | 后端包管理(https://docs.astral.sh/uv/) |
| Node.js | 20+ | 前端构建(推荐 22) |
| PostgreSQL | 15+ | 生产数据库(可换,见 §5) |
| Redis | 7+ | Celery broker / result backend |
| Docker + Compose | 24+ | 仅 Docker 模式需要 |

约定:后端命令用 `uv run ...`,前端用 `npm ...`。仓库根目录有 `Makefile` 封装常用命令(`make help` 查看)。

### 1.2 资源要求(开发 / 测试 / 生产)

全栈由 11 个服务构成(`migrate` 为一次性任务,跑完退出):postgres、redis、api、worker、beat、flower(可选)、frontend、prometheus、alertmanager、grafana。下面按环境给最小与建议规格。

> **compose 仅供开发/联调**:`docker-compose.yml` 文件头已注明"生产部署另行裁剪,勿直接用于线上"。生产档位按"能上线的最小合理配置"给出,而非照搬单机 compose。

#### 开发环境(单机起全栈)

一台机器 `make up` 拉起全部服务,够写代码与联调。

| 资源 | 最小 | 建议 |
|------|------|------|
| CPU | 2 核 | 4 核 |
| 内存 | 6 GB | 8 GB |
| 磁盘 | 20 GB | 30 GB |

内存卡点:11 个容器空载约 3–4 GB,叠加 `docker build`(前端 `npm ci && build`)与 worker 多进程后按 6 GB 起。低于 4 GB 易在构建或 worker 启动时 OOM。

#### 测试环境(CI / 联调 / 预发)

| 资源 | 最小 | 建议 |
|------|------|------|
| CPU | 2 核 | 4 核 |
| 内存 | 4 GB | 8 GB |
| 磁盘 | 20 GB | 40 GB |

- 纯跑后端测试套件(内存 SQLite)2 核 / 2 GB 即可,无需起 Prometheus 全家桶。
- 若需拉起完整 compose 做集成验证,按 4 GB 起。
- 建议此档就固定 worker 并发:`celery worker --concurrency=2`,避免 CI 机核多时 prefork 派生过多子进程吃满内存。

#### 生产环境

生产不建议单机 compose 裸跑,分两种形态:

**形态 A —— 单机小规模(纳管 ≤ 20 台机器)**

| 资源 | 最小 | 建议 |
|------|------|------|
| CPU | 4 核 | 8 核 |
| 内存 | 8 GB | 16 GB |
| 磁盘 | 50 GB SSD | 100 GB SSD |

单容器资源限额建议(compose `deploy.resources.limits`):

| 服务 | CPU | 内存 |
|------|-----|------|
| postgres | 1–2 | 2 GB |
| redis | 0.5 | 1 GB(设 `maxmemory`) |
| api(×2 副本) | 各 0.5 | 各 512 MB |
| worker | 1–2 | 1–2 GB(须 `--concurrency=2~4` 定死) |
| beat | 0.2 | 256 MB |
| prometheus | 1 | 2 GB(随纳管节点数增长) |
| grafana | 0.5 | 512 MB |
| alertmanager / frontend / flower | 各 0.2 | 各 128–256 MB |

**形态 B —— 可扩展生产(纳管 > 50 台或要求高可用)**

- **数据层外置**:Postgres、Redis 改用云托管(RDS / ElastiCache),具备备份、主从与监控;不塞进 compose。
- **api / worker 水平扩**:两者无状态,api 起 2+ 副本,worker 按队列深度扩。
- **beat 恒为单例**:多实例会重复触发定时任务。
- **监控栈独立部署**:Prometheus 磁盘按 `保留天数 × 节点数 × series` 估算,单列数据盘,避免与业务抢 IO。
- **整体起步**:控制面节点 8–16 GB / 4–8 核,数据层与监控栈另行计算。

#### 生产必查三项

1. **worker 并发未设默认吃满 CPU**:代码未配 `--concurrency`,Celery prefork 默认等于容器可见核数。8 核机上单 worker 会派生 8 个子进程(每个约 150–250 MB),仅此一项就吃 1.5–2 GB。生产务必显式设并发。
2. **Prometheus 磁盘随节点线性增长**:默认未配 `--storage.tsdb.retention.time`,数据会持续累积。生产要设保留期(如 15d/30d)并挂独立数据盘。
3. **默认口令/密钥必须替换**:compose 中 `POSTGRES_PASSWORD`、`GRAFANA_PASSWORD` 为弱默认值,`YIMAI_JWT_SECRET` 为占位串,`YIMAI_SECRET_MASTER_KEY` 默认空。生产须全部经环境变量 / 密钥管理注入真实值(安全红线)。

## 2. 本地开发起步

适合改代码、调试。数据库可用本地 PostgreSQL,或直接用 SQLite 零依赖起步(见 §5)。

```bash
# 1. 准备环境变量
cp .env.example .env          # 按需修改;本地最省事可把 YIMAI_DATABASE_URL 改成 sqlite

# 2. 后端
cd backend
uv sync                        # 装依赖(读 uv.lock,可复现)
uv run alembic upgrade head    # 建表
uv run python -m app.cli.seed  # 灌种子管理员(默认 admin/admin)
uv run uvicorn app.main:app --reload --port 8000

# 3. 前端(另开终端)
cd frontend
npm install
npm run dev                    # Vite dev server,默认 http://localhost:5173
```

需要异步任务(部署/回滚/状态采集)时,另起 Celery:

```bash
cd backend
uv run celery -A app.workers.celery_app worker --loglevel=info   # worker
uv run celery -A app.workers.celery_app beat --loglevel=info     # 定时(状态采集/轮询兜底)
```

用 Makefile 更省事:`make backend-dev` / `make frontend-dev` / `make migrate` / `make seed`。

访问:前端 http://localhost:5173,后端健康检查 http://localhost:8000/healthz,OpenAPI 文档 http://localhost:8000/docs。

## 3. 手动打包与部署(不依赖 Docker)

适合把产物部署到一台已装好 Python/Node/数据库的服务器上。

### 3.1 后端打包

后端是纯 Python 应用,「打包」= 同步依赖 + 带上代码与迁移。有两种做法:

**做法 A:目标机用 uv 同步(推荐,可复现)**

```bash
# 在目标机上
cd backend
uv sync --frozen --no-dev      # 严格按锁文件装生产依赖(不装测试/lint 工具)
uv run alembic upgrade head    # 建表 / 升级到最新
uv run python -m app.cli.seed  # 首次部署灌管理员
```

**做法 B:构建 wheel 分发**

```bash
cd backend
uv build                       # 产出 dist/*.whl
# 目标机:python -m venv .venv && .venv/bin/pip install 该whl + uvicorn[standard] celery
```

生产用 gunicorn 管理 uvicorn worker(比裸 uvicorn 稳):

```bash
uv run gunicorn app.main:app \
  -k uvicorn.workers.UvicornWorker \
  -w 4 -b 0.0.0.0:8000
```

Celery worker/beat 各起一个进程(建议用 systemd 或 supervisor 托管):

```bash
uv run celery -A app.workers.celery_app worker --loglevel=info
uv run celery -A app.workers.celery_app beat  --loglevel=info
```

关键环境变量(通过 systemd `Environment=` 或 `.env` 注入):`YIMAI_DATABASE_URL`、`YIMAI_REDIS_URL`、`YIMAI_JWT_SECRET`(32字节+)、`YIMAI_SECRET_MASTER_KEY`(KMS 注入)。

> systemd 单元示例(`/etc/systemd/system/axon-api.service`):
> ```ini
> [Service]
> WorkingDirectory=/opt/axon/backend
> EnvironmentFile=/opt/axon/.env
> ExecStart=/opt/axon/backend/.venv/bin/gunicorn app.main:app -k uvicorn.workers.UvicornWorker -w 4 -b 0.0.0.0:8000
> Restart=always
> ```

### 3.2 前端打包

前端是静态 SPA,`npm run build` 产出纯静态文件,交给任意静态服务器/nginx 托管。

```bash
cd frontend
npm ci                         # 按 package-lock.json 装(可复现)
npm run build                  # 产出 dist/(静态资源)
```

把 `dist/` 部署到 nginx。参考 `frontend/nginx.conf`(已含 SPA 路由回退 + `/api`、`/ws` 反代到后端)。手动部署时把其中 `proxy_pass` 指向你的后端地址,再:

```bash
cp -r dist/* /var/www/axon/
# nginx 配置 root /var/www/axon; 并按 nginx.conf 设置 try_files 与反代
```

### 3.3 监控组件(可选,手动部署)

Prometheus / Alertmanager / Grafana 用官方二进制或包管理器装。配置直接复用仓库 `ops/` 下的文件:

- `ops/prometheus/prometheus.yml` → Prometheus 配置(含 file_sd 抓取 `ops/prometheus/targets/nodes.json`,由后端 T1.13 自动写入)
- `ops/prometheus/rules/node_alerts.yml` → 告警规则
- `ops/alertmanager/alertmanager.yml` → Alertmanager(webhook 指向 `控制面/api/webhooks/alert`)
- `ops/grafana/provisioning/` → Grafana 数据源与面板预置

## 4. Docker 模式打包与部署

仓库已提供两个 Dockerfile(`backend/Dockerfile`、`frontend/Dockerfile`)与根目录 `docker-compose.yml`,一键拉起全栈。

### 4.1 一键全栈(开发/联调最快)

```bash
cp .env.example .env           # 按需改端口、密码、密钥
docker compose up -d --build   # 或 make up
docker compose ps              # 看状态
docker compose logs -f api     # 跟随后端日志
```

compose 启动顺序已编排:`migrate` 服务先跑 `alembic upgrade head` 建表,紧接着 `python -m app.cli.seed` 灌种子管理员(默认 admin/admin,可在 `.env` 用 `YIMAI_SEED_ADMIN_USER` / `YIMAI_SEED_ADMIN_PASSWORD` 覆盖);api/worker/beat 等其成功后再启,避免建表竞态。seed 幂等(账号已存在则跳过),故 Docker 模式首次部署即自带管理员账号,无需手动灌。

访问:前端 http://localhost:5173,后端 http://localhost:8000,Flower http://localhost:5555,Prometheus http://localhost:9090,Grafana http://localhost:3000(admin/admin),Alertmanager http://localhost:9093。

停止:`docker compose down`(加 `-v` 连数据卷一起删)。

> 注意:仓库目录名含中文,compose 无法从目录名推导 project name,故 `docker-compose.yml` 显式写了 `name: yimai-axon`。勿删该字段。

### 4.2 单独构建镜像(用于推送镜像仓库 / 生产部署)

```bash
# 后端镜像(uv 装依赖 + uvicorn 起服务;worker/beat/flower 由 compose command 覆写)
docker build -t your-registry/axon-backend:1.0.0 ./backend

# 前端镜像(多阶段:node 构建 → nginx 托管产物)
docker build -t your-registry/axon-frontend:1.0.0 ./frontend

# 推送
docker push your-registry/axon-backend:1.0.0
docker push your-registry/axon-frontend:1.0.0
```

同一个后端镜像跑不同角色,靠覆写 `command`:

| 角色 | command |
|------|---------|
| API | `uvicorn app.main:app --host 0.0.0.0 --port 8000` |
| worker | `celery -A app.workers.celery_app worker --loglevel=info` |
| beat | `celery -A app.workers.celery_app beat --loglevel=info` |
| flower | `celery -A app.workers.celery_app flower --port=5555` |
| 迁移(一次性) | `alembic upgrade head` |

### 4.3 生产部署要点

- **镜像里不塞密钥**:`YIMAI_JWT_SECRET`、`YIMAI_SECRET_MASTER_KEY`、`YIMAI_WEBHOOK_SECRETS` 等通过部署平台的 secret 机制注入(K8s Secret / compose `environment` / systemd EnvironmentFile),勿写进镜像或提交进仓库。
- **迁移单独跑**:生产用一次性 Job(compose 的 `migrate` 服务 / K8s Job)执行 `alembic upgrade head`,不要让每个 api 副本都建表。
- **数据库/Redis 用托管服务**:生产别用 compose 里的 postgres/redis 容器(无持久化保障),改用云托管实例,只需改 `YIMAI_DATABASE_URL` / `YIMAI_REDIS_URL`。
- **前端反代**:`frontend/nginx.conf` 里的 `/api`、`/ws` 反代目标改成生产后端地址(或 K8s Service 名)。

## 5. 切换数据库

系统用 SQLAlchemy 2.0(async)+ Alembic,数据库通过**一个环境变量** `YIMAI_DATABASE_URL` 切换,代码无需改动。引擎按方言自动适配(`backend/app/core/db.py`:SQLite 不传连接池参数,其余走连接池 + pre-ping)。

### 5.1 支持矩阵

| 数据库 | DSN 示例 | 驱动依赖 | 状态 |
|--------|----------|----------|------|
| PostgreSQL(默认/生产推荐) | `postgresql+asyncpg://用户:密码@主机:5432/库名` | `asyncpg`(已装) | ✅ 一等公民 |
| SQLite(本地/试用) | `sqlite+aiosqlite:///./axon.db` | `aiosqlite`(已装) | ✅ 开箱即用 |
| MySQL / MariaDB | `mysql+aiomysql://用户:密码@主机:3306/库名` | `aiomysql`(需自行加) | ⚠️ 需装驱动并验证 |

迁移已做跨方言处理:JSON 字段用 `sa.JSON().with_variant(JSONB, "postgresql")`(PG 走 JSONB,其余走通用 JSON),Enum 类型清理带方言判断。因此 SQLite/MySQL 也能 `alembic upgrade head`。

### 5.2 切到 SQLite(最省事的本地/试用)

```bash
# .env 或环境变量
YIMAI_DATABASE_URL=sqlite+aiosqlite:///./axon.db
```

```bash
cd backend
uv run alembic upgrade head
uv run python -m app.cli.seed
uv run uvicorn app.main:app --reload
```

SQLite 是单文件(`backend/axon.db`),适合本地开发和演示。**不建议生产用**:并发写弱、无网络访问。

### 5.3 切到 PostgreSQL(生产默认)

```bash
YIMAI_DATABASE_URL=postgresql+asyncpg://yimai:强密码@db.host:5432/yimai
```

先在目标 PG 建好库和账号,再 `alembic upgrade head`。连接池参数可调(`YIMAI_DB_POOL_SIZE` 等,见 config.py)。

### 5.4 切到 MySQL / MariaDB

MySQL 未随包安装驱动,需自行添加:

```bash
cd backend
uv add aiomysql            # 装异步 MySQL 驱动
```

```bash
YIMAI_DATABASE_URL=mysql+aiomysql://用户:密码@主机:3306/库名?charset=utf8mb4
```

然后 `alembic upgrade head`。建议先在测试库跑一遍全部迁移的升/降级,确认无方言兼容问题(尤其 Enum、JSON、字符串长度上限)再上生产。

### 5.5 切库注意事项

- **数据不会自动迁移**:切换 DSN 只是指向新库,旧库数据不会搬过去。需要迁移数据得自己导出/导入。
- **切库后必跑迁移**:新库是空的,先 `alembic upgrade head` 建表,再 `seed` 灌管理员。
- **保险箱主密钥要一致**:`YIMAI_SECRET_MASTER_KEY` 变了会导致已存的加密凭证(SSH 私钥、平台 token)解不开。换库时如果沿用旧数据,主密钥必须保持一致。
- **测试用内存 SQLite**:测试固定用 `sqlite+aiosqlite:///:memory:`,与生产库隔离,不受此配置影响。

## 6. 首次初始化清单

无论哪种部署方式,首次上线按序做:

1. 准备 `.env`(至少设好 `YIMAI_DATABASE_URL`、`YIMAI_JWT_SECRET`、`YIMAI_SECRET_MASTER_KEY`;如需自定义初始管理员,设 `YIMAI_SEED_ADMIN_USER` / `YIMAI_SEED_ADMIN_PASSWORD`)
2. `alembic upgrade head` —— 建表
3. `python -m app.cli.seed` —— 创建管理员(默认 admin/admin,**登录后立即改密**)。**Docker 模式无需手动执行**:`migrate` 服务已自动"建表 + 灌种子",seed 幂等可重复跑
4. 起 api + worker + beat(+ flower 可选)
5. 起前端(nginx 托管 dist,反代指向后端)
6. 起监控三件套(Prometheus/Alertmanager/Grafana),确认 Grafana 能看到 node_exporter 指标
7. 登录前端,验证服务器/服务/部署/配置/监控/告警各页可用

## 7. 关键运维流程速查

本节覆盖 Epic 2/3 的核心运维动作。所有写操作异步落 task,前端轮询到终态回显;所有 prod 高危操作走审批(见 7.6)。

### 7.1 触发部署 + 发布策略

部署页选服务 → 「触发部署」→ 填版本 + 选策略 → 提交。

| 策略 | k8s 实现 | 裸机(systemd/docker)实现 |
|------|----------|--------------------------|
| rolling(滚动) | 原生 `rollout restart` | 逐 placement 分批重启 |
| recreate(重建) | scale 0 → scale 回目标副本 | 全停 → 全起(有停机窗口) |
| canary / blue-green | ⚠️ 需 Argo Rollouts / service mesh,当前返回 501(未接入) | ⚠️ 同左,未接入 |

> canary/蓝绿是有意的"明确未实现"而非静默失败:选了会收到 501 提示,不会伪装成功。接入 Argo Rollouts 后在 `app/services/release_strategy.py` 补实现即可,上层不改。

### 7.2 配置版本化与下发到目标机

配置页选服务:
1. **新建版本**:填内容 + 格式 + **下发路径**(target_path,如 `/etc/app/app.env`);敏感值用 `${secret:credential_id}` 占位,下发时从保险箱注入,库里不落明文。
2. **下发**:点某版本「下发」→ 异步写到该服务所有 placement 的 target_path,按 `reload_mode` reload 或 restart 生效 → 逐目标落 `config_deliveries` 记录。
3. **查看下发结果**:点「下发结果」看每个目标机 success/failed(支持部分成功)。
4. **配置回滚**:切换生效版到历史某版(activate),再下发即可。

> 无 target_path 的版本只能查看不能下发(不知道写到哪)。

### 7.3 一键回滚

部署页「一键回滚」→ 二次确认 → 重部署上一次成功的制品,生成新记录,被回滚版落 `rolled_back`(闭环可追溯)。

### 7.4 环境晋升(promotion)

把 staging 验证过的**同一制品**搬到 prod(不重构建):调 `POST /api/services/{prod服务id}/promote`,body 带 `source_service_id`(staging 服务 id)。源与目标须同名不同 env。

### 7.5 全链路关联与部署标注

- **扫描门禁**:部署带 git_sha 时,`deploy_block_on_critical` 开启则命中 critical 漏洞拦截(422)。
- **部署详情**:点部署记录看关联的扫描结论(扫过没、有无高危、报告链接)。
- **部署标注**:监控页选一个服务,曲线上按部署时间点打竖线,看"发布后 CPU 涨没涨"(§9.2)。

### 7.6 生产审批流

`require_prod_approval`(默认开)开启时,prod 的 deploy 不直接执行,先落 pending 审批:
1. 发起人触发 prod 部署 → 返回 `approval_id`(pending)。
2. 审批页:**具 approval 权限的另一人**(四眼原则,不能批自己发起的)approve/reject。
3. approve → 建 task 走与直接部署完全一致的编排;reject → 关闭,记录理由。全程入审计。

### 7.7 告警联动与通知

- Alertmanager → `POST /api/webhooks/alert`(HMAC 验签)→ 落库 + 主页展示。
- **自动回滚**:`auto_rollback_on_alert`(默认关)开启时,critical+firing+可定位服务的告警触发自动回滚。
- **通知触达**:配了 `YIMAI_NOTIFY_WEBHOOK_URL` 则 firing 告警推送到 IM(钉钉/飞书/企微/Slack 自定义机器人的 webhook);未配则静默。

### 7.8 部署回流与轮询兜底

- **回流**:CI 流水线跑完 `POST /api/webhooks/deployment`(HMAC + 幂等键 `pipeline_id+service+env`)→ 主页 feed 实时更新。
- **轮询兜底**:webhook 整条丢失时,Celery beat 定时(`deploy_reconcile_interval_sec`)扫仍 running 的部署,查 CI 当前状态补齐终态。⚠️ 需生产 pipeline provider 工厂(当前 `app/workers/deploy_tasks.py` 标注了此缺口,无 provider 时优雅跳过)。

### 7.9 Agent 接入(Epic 4,控制面侧就绪)

- 服务器 `access_mode=agent` 的操作经 `AgentGateway` 走连接管理器下发命令、等 result ACK(两段 ACK)。
- 断连一致性:超时判 unknown(不武断 failed)、prod 高危离线直接拒绝、fencing token 防 SSH fallback 双执行。
- ⚠️ **未接入**:Go Agent 本体二进制(T4.2)与真实 gRPC wire 未实现;当前 transport 是可注入抽象,接真实 gRPC 时上层零改动。无 manager 注入时 AgentGateway 为占位(返回 501)。

## 8. 常见问题

**Q: `alembic upgrade head` 报连不上库?**
检查 `YIMAI_DATABASE_URL` 是否正确、库是否已建、账号权限是否够。健康检查 `/healthz` 含 DB 探活,可先看它。

**Q: webhook(部署/扫描/告警)全返回 401?**
入向 webhook 走 HMAC 签名(§8.3),需在 `YIMAI_WEBHOOK_SECRETS` 里为每个来源配 secret。缺省为空 = 全拒。

**Q: 部署/回滚一直不动?**
异步动作靠 Celery worker 执行,确认 worker 进程在跑、Redis 连得上。Flower(:5555)可观测任务。

**Q: 前端页面能开但接口全 404 / CORS 报错?**
本地开发:确认后端在 8000 起着。生产:确认 nginx `/api`、`/ws` 反代目标正确。CORS 白名单由 `YIMAI_CORS_ORIGINS` 控制。

**Q: 想换掉默认端口?**
Docker 模式改 `.env` 里的 `*_PORT`;手动模式改各自启动命令的 `--port` 与 nginx 配置。
