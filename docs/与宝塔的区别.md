# 与宝塔面板的区别

> 目的:说明本项目(统一运维控制面)与宝塔面板(BT Panel / aaPanel)在定位、能力和适用场景上的差异,帮助判断两者是竞品还是可以叠加使用。

## 一句话概括

- **宝塔**:装在**单台服务器**上的**建站 / 主机管理面板**,把这台机器上的 LNMP/LAMP、网站、数据库、FTP、证书等用图形界面包起来,让你不写命令行也能把网站跑起来、把主机管好。
- **统一运维控制面**:一个**多机、跨环境**的中央**控制面(Control Plane)**,核心是对接现有平台(扫描 / 流水线 / K8s / Prometheus 等),把**代码扫描 → 流水线部署 → 部署后操作 → 运行监控**用关联键串成一条**可回溯的交付治理链路**。核心原则是**不重造任何一个平台的功能,只做编排、关联和统一操作入口**。

## 根本差异

宝塔解决的是"**一台机器怎么方便地跑起网站、管好主机**",受众是站长和小团队,重点在**单机资源的图形化管理**。

本项目解决的是"**一批服务从代码到生产怎么安全、可控、可追溯地交付和运维**",受众是有多环境、多机器、需要发布管控和审计的研发 / 运维团队,重点在**跨机、跨平台的交付链路治理与关联**。

简单说:宝塔是"服务器保姆",本项目是"交付流水线的控制塔"。

## 逐维度对比

| 维度 | 宝塔面板 | 统一运维控制面 |
|---|---|---|
| **管理范围** | 单台服务器(面板即那台机器的图形化管家) | 中央控制台纳管**多台服务器 / 多个服务**,经 SSH 或 Agent 下发操作 |
| **核心场景** | 建站托管:网站、MySQL、PHP 版本、FTP、文件管理器、SSL 证书、计划任务 | 应用**交付生命周期治理**:触发部署、配置版本管理与下发、发布策略、回滚 |
| **CI/CD 接入** | 基本没有,靠手动上传或 git 拉取 | 对接 Jenkins / GitLab 流水线;入向 Webhook(HMAC 验签 + 时间窗 + 幂等) |
| **发布能力** | 无发布策略概念 | rolling / recreate 已实现;审批流(四眼原则,生产默认需审批)、环境晋升 |
| **质量门禁** | 无 | 扫描结果回流 + critical 问题拦截部署(质量门禁) |
| **监控与告警** | 面板自带简单负载图 | Prometheus + node_exporter 自举 + Grafana;PromQL 白名单代理;告警驱动**自动回滚** + 发布后健康检查 |
| **运行时支持** | 主要面向裸机进程 | systemd / Docker / K8s 三种运行时统一适配,底层自动路由 |
| **可回溯性** | 弱 | 全链路关联:一次异常可回溯到"哪次提交、谁点的部署、扫描报没报过问题",操作全程审计落库 |
| **权限模型** | 面板账号 | 按环境分级鉴权(生产操作需审批),操作审计可追溯 |
| **定位类比** | 主机 / 建站面板 | DevOps / SRE 的交付与运维治理平台 |

## 两者的关系

二者不是非此即彼,甚至可以叠加:

- **宝塔** 负责单机的基础环境:装环境、跑网站、管数据库和证书。
- **统一运维控制面** 负责跨机的交付与治理:谁能部署、按什么策略发布、出问题怎么回溯和回滚。

如果只是在一台服务器上跑几个网站,宝塔足够;当服务分布在多台机器、跨多个环境、需要发布管控和全链路追溯时,才需要本项目这类控制面。
