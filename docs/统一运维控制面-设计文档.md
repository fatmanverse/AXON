## 1. 背景与目标

### 1.1 现状痛点

当前市面上没有一个平台能在**单一入口**完成以下全部操作,团队被迫在多个系统间来回切换:

- 代码扫描结果在 SonarQube 里看
- 流水线在 Jenkins / GitLab CI 里跑
- 服务启停、部署在 K8s / 各台服务器上零散操作
- 监控在 Grafana 里看
- 出了问题,无法把"线上一个异常"一路回溯到"是哪次提交、谁点的部署、扫描报没报过问题"

**根本矛盾**:扫描 / 流水线属于"交付态",服务操作 / 监控属于"运行态",两者的数据模型和权限模型天然割裂。现有一体化平台在"运行态"这一环普遍做得浅。

### 1.2 目标

构建一个统一控制面,实现:

1. **统一纳管**:K8s、Docker、裸机/VM(systemd/进程)上的服务,统一列出、统一操作。
2. **统一操作**:启停、重启、删除、部署、回滚、配置更新,一套按钮,底层自动路由。
3. **交付态接入**:接入代码扫描、流水线,部署事件自动回流到主页。
4. **运行态可观测**:服务器状态(Agent)、资源指标(Prometheus)、告警(Alertmanager)统一展示。
5. **全链路关联**:通过关联键,把一次提交 → 扫描 → 流水线 → 部署 → 运行监控串成一条可回溯的链路。

### 1.3 非目标

- 不重写扫描引擎、CI 引擎、监控存储,这些用成熟开源件当"发动机"。
- 不追求一次做全,分期演进(见 [MVP 分期](#17-mvp-分期规划))。

## 2. 设计原则

| 原则 | 含义 |
|------|------|
| **控制面而非平台** | 你写的是编排层、关联层、统一入口,不是四块功能本身。 |
| **Adapter 屏蔽差异** | 每个外部平台(扫描/CI/监控/运行时)包一个 Adapter,换工具不动上层。 |
| **控制通道与指标通道分离** | 下命令走 SSH/Agent(点对点、要审计);看指标走 Prometheus(时序、聚合)。两者不混。 |
| **一切写操作异步化** | 部署、回滚、扩缩容都是长任务,落 job → 队列 → 异步执行 → UI 轮询/推送。 |
| **关联键是灵魂** | `service` / `git_sha` / `deployment_id` 三键贯穿全链路。 |
| **统一模型对上,多态执行对下** | UI 只见统一的「服务/服务器」模型,底层 systemd/docker/k8s 差异由执行器屏蔽。 |

### 3.2 后端模块划分

1. **接入管理** — 添加服务器、选 SSH/Agent、连通性测试、Agent 安装引导。
2. **执行器层** — `SSHExecutor` 与 `AgentGateway` 两实现,统一接口 `exec()` / `deploy()` / `updateConfig()`。
3. **服务模型 & 生命周期** — start/stop/restart/delete/deploy/rollback,屏蔽底层 systemd/docker/k8s。
4. **配置管理** — 配置版本库(留版本、diff、一键回滚)+ 下发 + reload/重启标注。
5. **监控聚合** — 服务状态(Agent 心跳)+ 资源指标(Prometheus)统一喂给 UI。
6. **交付态接入** — 扫描 Adapter、CI Adapter、Webhook 接收。
7. **安全底座** — RBAC、凭证保险箱、全量审计日志。
8. **异步任务队列** — 写操作落 job、异步执行、UI 轮询/推送进度。

## 4. 核心抽象模型

不管底层是 K8s、Docker 还是裸机,对上层只暴露一套统一的「服务」模型:

```
Service(统一模型)
├── 生命周期: start / stop / restart / delete
├── 部署:     deploy(拉新版本) / rollback
├── 配置:     getConfig / updateConfig / reloadConfig
├── 观测:     status / logs / metrics
└── runtime:  k8s | docker | systemd | process | cloud-fn
```

- `runtime` 字段决定动作路由到哪个执行器。
- UI 上用户点的都是同一个"重启"按钮,底层是 `kubectl rollout restart` / `systemctl restart` / `docker restart`,由 Adapter 屏蔽。
- 同理,每台服务器有 `access_mode` 字段(`ssh` / `agent`),上层无感。

## 5. 接入模式:SSH 与 Agent

平台**同时支持**两种接入通道,统一抽象在 `access_mode` 字段下。

| 维度 | SSH 模式(Agentless) | Agent 模式 |
|------|------|------|
| 目标机器是否装东西 | 不装 | 装一个自研 Agent |
| 下命令方式 | 控制面主动 SSH 进去执行 | 控制面 → Agent(Agent 本地执行) |
| 服务状态 | 控制面定时 SSH 拉取 | Agent 主动心跳上报(实时) |
| 网络要求 | 控制面必须能直连每台机器 | 机器可在内网,Agent 主动外连控制面 |
| 审计 | 弱(需额外记录) | 强(命令经 Agent,天然可审计) |
| 适用场景 | 机器少、同内网、快速接入 | 机器多、跨网、要实时状态和审计 |

### 5.1 统一执行器接口

```
interface Executor {
    exec(cmd)                // 执行命令
    deploy(artifact, spec)   // 部署
    updateConfig(cfg)        // 下发配置
    getServiceStatus()       // 拉服务状态
}

SSHExecutor  implements Executor   // 底层: Ansible / x/crypto/ssh
AgentGateway implements Executor   // 底层: gRPC 双向流 / WebSocket
```

同一个"重启服务"业务逻辑,根据服务器的 `access_mode` 自动路由到对应执行器。**从 SSH 平滑升级到 Agent 时,UI 和业务逻辑一行不改。**

### 5.2 Agent 设计要点

- **主动外连**:Agent → 控制面建长连(gRPC stream 或 WebSocket),避免要求机器开入站端口,天然支持内网。
- **三件事**:①心跳 + 状态上报 ②接命令并执行 ③接配置并下发/reload。
- **自举能力**:首次接入能自动安装/拉起 node_exporter,免去手动装。
- **远程升级**:Agent 自身能被控制面远程升级(几百台机器手动升级是灾难)。
- **降级兜底**:Agent 挂了能 fallback 到 SSH,保证不失联。

### 5.3 落地策略

**MVP 用 SSH 起步,架构预留 Agent 接口。** Agent 是能吃掉半个项目工作量的大件,早期用 SSH(配合 Ansible)快速验证价值;待机器规模、内网穿透、实时状态成刚需时,再切 Agent,而统一「服务」模型不用动。

### 5.4 断连一致性与脑裂防护(设计细化)

命令下发途中 Agent 断连,是分布式控制面的核心难题:控制面无法区分"命令没送达""送达没执行""执行了没回结果",task 会卡在 `running`。这里明确一套 **at-least-once + 幂等** 的语义(不追求成本高昂的 exactly-once)。`ServerCommand.task_id`(见 §15.5)是整套机制的幂等基石。

**① 两段 ACK,下发≠成功。**

- Agent 收到命令回 `received` ACK,执行完回 `result` ACK。
- 控制面依据 ACK 推进任务状态机,**绝不"下发即标记成功"**。

**② Agent 侧持久化 + 重连补报。**

- Agent 执行命令前先落盘 `task_id + 状态`,重连后 **主动补报** 所有未确认结果。
- 控制面按 `task_id` 幂等接收,重复结果直接丢弃。

**③ 命令尽量设计成幂等。**

- `deploy(artifact=X)`、`restart` 天然近幂等,重连补执行不会造成二次伤害。
- 非幂等操作(如"扩容 +1")改写成 **声明式**(如"目标副本=N"),规避重复执行的累加错误。

**④ 超时语义:`unknown` 而非 `failed`。**

- 命令超时后标记为 **`unknown`**(可能已成功执行),等重连核对或人工介入,**不可武断判失败**。
- `tasks` 表状态枚举因此新增 `unknown`(见 §14.6)。

**⑤ 断连期命令策略,按环境与危险度分档。**

| 操作 | Agent 离线时策略 |
|------|------|
| 生产高危(prod deploy / delete) | **直接拒绝并提示**,不排队——避免 Agent 数小时后重连突然执行过期的生产变更 |
| 低危(status 拉取、非生产 restart) | 可短暂排队 + **TTL 过期作废** |

**⑥ 脑裂防护:fencing token / 租约。**

> "控制面连不上 Agent" ≠ "Agent 已死"。可能只是网络分区,Agent 仍存活甚至正在执行。此时若 §5.2 的 SSH fallback 又下发一遍,就 **双重执行** 了。

- 同一 service 的写操作持有一个 **单调递增的 fence token**。
- SSH 通道执行前必须确认 Agent 租约已过期(fence 失效),**保证任一时刻只有一个通道能动同一目标**。

**⑦ 重连风暴防护。**

- 控制面重启时几百个 Agent 同时重连会打爆。Agent 重连必须 **指数退避 + jitter**。

## 6. 监控体系

**两条独立通道,职责不重叠。**

### 6.1 通道一:服务器 / 服务状态(通过 Agent)

Agent 定时心跳,上报:
- 机器在线状态、Agent 版本
- 该机器纳管服务的运行状态(running / stopped / 异常)
- 最近一次命令执行结果

SSH 模式下由控制面定时 SSH 探测补齐(实时性略差)。

### 6.2 通道二:资源监控(通过 Prometheus Agent = node_exporter)

- 每台机器装 **node_exporter**,由中心 **Prometheus** 抓取 CPU / 内存 / 磁盘 / 网络 / 负载。
- 控制面**不自己存指标**,而是查询 Prometheus 的 PromQL API,把数据画到自己 UI(或内嵌 Grafana 面板)。

> **职责分工**:自研 Agent 负责"控制 + 服务级状态";node_exporter 负责"主机资源指标"。Agent 可顺带自动部署和拉起 node_exporter。

### 6.3 告警(Alertmanager)

- Prometheus 告警规则触发 → Alertmanager → webhook 回调控制面 `/api/webhooks/alert`。
- 控制面在主页显示告警,并可联动:通知(钉钉/飞书/企微)+(高级)告警自动回滚。

## 7. 代码扫描接入

### 7.1 接入方式

与流水线一致,采用 **webhook 上报**。扫描工具(SonarQube / Semgrep / Trivy)扫描完成后,回调控制面:

```
POST /api/webhooks/scan
{
  "service": "order-service",
  "git_sha": "abc123...",
  "scanner": "sonarqube",
  "critical": 2,
  "high": 5,
  "medium": 12,
  "report_url": "https://sonar.xxx/dashboard?id=order-service",
  "passed": false
}
```

### 7.2 关联与卡点

- 扫描结果按 `git_sha` 挂到链路上。
- **部署卡点(quality gate)**:部署时可配置策略——"存在 critical 漏洞则拦截部署",在部署前查 `scan_results` 表。
- 点开一次部署,能看到"这次上线的代码扫描过没有、有没有高危"。

## 8. 流水线接入

流水线(Jenkins / GitLab CI / Argo CD / Tekton)与控制面有**双向关系,两个方向都要支持**。

### 8.1 模式 A:控制面「触发」流水线(主动,出向)

用户在 UI 点"部署服务 X 到生产" → 控制面调流水线 API 触发。

```
UI 点部署 → 控制面 → 调 Jenkins/GitLab API → 流水线跑起来
                 ↓
           落一条 deployment 记录(status=running)
```

各 CI 的触发接口(由 `PipelineAdapter` 统一封装 `trigger()` / `getStatus()` / `getLogs()`):
- Jenkins:`POST /job/xxx/buildWithParameters`
- GitLab CI:`POST /projects/:id/pipeline`(trigger token)
- Argo CD:`POST /api/v1/applications/{app}/sync`

### 8.2 模式 B:流水线「通知」控制面(被动,入向)—— 更重要

不是所有部署都从 UI 发起。开发者直接 push 代码可能触发了自动部署,控制面必须**被动感知**,否则主页信息是错的。

| 办法 | 说明 | 优劣 |
|------|------|------|
| **Webhook(推荐)** | 流水线跑完回调 `/api/webhooks/deployment` | 实时、准确,需改流水线配置 |
| **轮询** | 控制面定时查 CI API 历史 | 不用改流水线,但有延迟、耗资源 |

**关键动作:在流水线最后加一个"上报步骤"(整个方案的胶水)。**

```yaml
# GitLab CI 示例:流水线最后一步
report-deployment:
  script:
    - |
      curl -X POST $CONTROL_PLANE/api/webhooks/deployment \
        -H "Authorization: Bearer $TOKEN" \
        -d '{
          "service": "order-service",
          "env": "prod",
          "git_sha": "'$CI_COMMIT_SHA'",
          "version": "'$CI_COMMIT_TAG'",
          "pipeline_id": "'$CI_PIPELINE_ID'",
          "pipeline_url": "'$CI_PIPELINE_URL'",
          "operator": "'$GITLAB_USER_LOGIN'",
          "status": "success",
          "artifact": "registry.xxx/order:'$CI_COMMIT_SHA'"
        }'
```

不管部署由 UI 触发还是别人 push 触发,**只要这一步执行,控制面就有了完整、准确的部署记录**。

### 8.3 Webhook 鉴权与幂等(设计细化)

§8.2 示例用共享 `Bearer $TOKEN`,草案够用,进生产有四个洞,逐一收敛。所有入向 webhook(`/deployment` `/scan` `/alert`)统一遵循本节约定。

**① 鉴权:共享 Bearer → 每源独立 HMAC 签名。**

- 共享 token 一旦泄漏(CI 日志/环境变量误打印)即门户洞开,且无法区分上报来源。
- 改为 **HMAC-SHA256 签名**:每个 webhook 源(每条 CI project、每个 scanner)分配独立 secret,对 **原始 request body** 签名,签名放 `X-Signature` header,服务端用对应源的 secret 验签。GitLab/GitHub 原生支持。
- 签名载荷带 `timestamp`,服务端校验时间窗(±5min)**防重放**。
- 叠加 **来源 IP 白名单** 作为纵深防御。
- 所有 secret 进凭证保险箱(§13),支持 **双 secret 并存的轮转过渡**。

**② 幂等:防重复记录。**

- webhook 天然 at-least-once(网络抖动、CI 超时重试),直接 INSERT 会产生重复记录。
- 约定 **幂等键**:deployment 用 `pipeline_id + service + env`(或由上报方带 `event_id`);scan 用 `git_sha + scanner`。
- DB 侧加 **唯一约束** + `INSERT ... ON CONFLICT DO UPDATE`,把"重复上报"收敛为"幂等更新"。

**③ 乱序保护。**

- 因重试,`running` 事件可能晚于 `success` 到达。
- deployment 状态机只允许 **前进不允许回退**(`running → success/failed`,禁止 `success → running`);用事件自带 `finished_at` 判新旧,旧事件丢弃。

**④ 上报失败与交付解耦。**

- CI 最后的上报步骤设 `allow_failure`(**不阻塞交付流水线变红**)。
- 保留 §8.2 提到的 **轮询兜底** 作为补偿通道,与 webhook 靠幂等键去重——这是"准确"与"解耦"的平衡点。

## 10. 多环境部署

### 10.1 环境模型

`env` 字段贯穿所有相关表(services / deployments / configs):

```
env: dev | staging | prod  (可扩展 pre / gray 等)
```

- 每个服务在不同环境是独立的部署实例,独立的配置、独立的版本。
- 主页和服务详情页按环境分列/切换,清晰看到"同一个服务在各环境跑的是哪个版本"。

### 10.2 环境差异化管控

| 环境 | 权限 | 操作要求 |
|------|------|----------|
| dev | 宽松,开发者可自助 | 直接操作 |
| staging | 中等 | 可自助,记录审计 |
| prod | 严格,仅授权角色 | **走审批流 + 二次确认 + 全量审计** |

### 10.3 环境晋升(promotion)

支持"把 staging 验证过的版本晋升到 prod":本质是取 staging 那条成功 deployment 的 artifact,在 prod 环境重新部署,保证上线的是同一个已验证制品(而非重新构建)。

## 11. 发布策略

生产不能直接全量替换。发布策略也进入统一模型,按 `runtime` 选择实现:

| 策略 | 说明 | K8s 实现 | 裸机实现 |
|------|------|----------|----------|
| **滚动(Rolling)** | 逐批替换旧实例 | Deployment 原生滚动 | Agent 分批重启 |
| **金丝雀(Canary)** | 先放小流量到新版本,观察后放全量 | Argo Rollouts + 服务网格 | Agent 分批 + 负载权重 |
| **蓝绿(Blue-Green)** | 新旧两套并存,切流量瞬时完成 | Service 切 selector | 负载均衡切上游 |
| **重建(Recreate)** | 停旧起新(有停机) | Deployment recreate | stop → deploy → start |

### 11.1 发布后健康检查

- 部署完成后自动跑**健康检查 / 就绪探针**(HTTP 探测 / 命令探测)。
- 检查失败 → 自动标记部署失败,并可触发**自动回滚**。
- 金丝雀阶段可结合 Prometheus 指标(错误率、延迟)做**自动分析**,不达标则中止发布。

### 11.2 回滚机制

- 手动回滚:UI 点回滚 → 读 `previous_deployment_id` → 重新部署上一版 artifact。
- 自动回滚:健康检查失败 / 告警触发 → 自动执行回滚流程。
- 回滚本身也是一次 deployment 记录(`status=rolled_back`),形成闭环、可审计。

---

## 12. 配置管理

配置更新不只是"改文件重启",要做成**可版本化、可回滚**。

### 12.1 配置版本库

- 控制面自己维护配置的版本库,每次改动留版本,支持 **diff** 与 **一键回滚**。
- 严禁直接去机器上改文件——否则改错无法回退。

### 12.2 下发方式

| 方式 | 适用 | 说明 |
|------|------|------|
| **推(push)** | 裸机 / systemd | 控制面 → 目标机,改文件 + reload/重启 |
| **拉(pull)** | 云原生服务 | 服务启动时来控制面/配置中心取(挂 ConfigMap) |

### 12.3 生效方式

`Service` 模型标注该服务配置变更后:
- **热更(reload)**:不断连,发信号重载。
- **重启生效**:必须重启进程。

初期做到:**配置版本化存储 + 下发 + 一键回滚**,先保住"改错能退回去"这条命脉,不做配置中心的推送长连接。

## 13. 安全底座

控制面握有所有平台的高权限凭证,是攻击者的天堂,安全不能省。

| 项 | 要求 |
|----|------|
| **凭证保险箱** | SSH 私钥、平台 token 进 Vault/KMS,不落库明文。 |
| **RBAC** | 自建权限模型,精确到"谁能点生产删除/部署"。 |
| **审批流** | 生产高危操作(部署/删除)进审批队列,二次确认 + 记录审批人。 |
| **全量审计** | 所有写操作(部署/启停/删除/改配置)不可篡改地落审计日志,可回放。 |
| **最小权限** | 各 Adapter 使用的平台账号按最小权限授予。 |
| **通知触达** | 关键操作、告警接钉钉/飞书/企微/Slack。 |

## 14. 数据模型(表设计)

> 关联键:`service` / `git_sha` / `deployment_id` 贯穿全表。

### 14.1 servers(服务器)

```
id                主键
name              服务器名
host / ip         地址
access_mode       ssh | agent
ssh_credential_id 指向凭证保险箱(ssh 模式)
agent_id          Agent 标识(agent 模式)
agent_status      online | offline | unknown
agent_version     Agent 版本
labels            标签(env、区域、机房等)
created_at / updated_at
```

### 14.2 services + service_placements(服务定义与放置,设计细化)

> **为何拆两张表**:原单表的 `server_id`(所在服务器)隐含"一个服务 = 一台机器上的一个东西",与 `runtime` 支持 K8s / 多副本 **自相矛盾**:K8s 服务不属于单台 server(多副本调度到多节点),裸机高可用服务也常部署到多台机器,单个 `server_id` 都表达不了。故将 **服务逻辑定义** 与 **服务放置(placement)** 分离;同时把 **期望态(desired)与观测态(observed)分开**。

**services(逻辑定义,不绑单机)**

```
id                主键
name              服务名(= 关联键 service)
env               dev | staging | prod
runtime           k8s | docker | systemd | process | cloud-fn
runtime_ref       JSON,承载 runtime 差异化的服务标识:
                    k8s     → {cluster_id, namespace, workload}
                    systemd → {unit_name}
                    docker  → {container_name}
desired_version        期望版本(声明式,替代原 current_version)
current_deployment_id  当前生效的部署
reload_mode       reload | restart(配置生效方式)
health_check      健康检查配置(HTTP/命令)
created_at / updated_at
```

**service_placements(放置,一对多,表达"跑在哪 / 实际什么状态")**

```
id                主键
service_id        指向 services
server_id         裸机/systemd 才填;k8s 留空(副本分布实时查 k8s API,不落库)
observed_version  该放置点实际观测到的运行版本
observed_status   running | stopped | error | unknown
last_seen_at      最近一次观测/心跳时间
```

要点:
- **desired vs observed 严格分离**:`services.desired_version` 是期望,`service_placements.observed_*` 是实际观测,二者不混。
- **聚合状态需明确定义**:多副本下 3/5 在跑新版本,聚合态是 `progressing` 还是 `degraded`,由聚合逻辑按 placement 计算,不落死字段。
- **K8s 副本数/Pod 分布实时查 K8s API,不落库**(落库必过期);placement 表主要对裸机多实例有意义。
- **service identity 因 runtime 而异**:k8s 用 `(cluster, ns, workload)`,裸机用 `(server, unit)`,统一由 `runtime_ref` JSON 承载,不硬塞进单一 `server_id` 列。

### 14.3 deployments(部署记录 —— 焊点)

```
id                    主键(= deployment_id 关联键)
service               关联键
env                   dev | staging | prod
git_sha               部署了哪次提交(关联键)
version               v1.2.3 / tag
artifact              镜像地址/包
strategy              rolling | canary | blue-green | recreate
source                ui-triggered | pipeline-webhook | manual
pipeline_id           回溯到哪条流水线
pipeline_url          一键跳转流水线
operator              操作人
status                running | success | failed | rolled_back
previous_deployment_id 上一次成功部署(支持一键回滚)
scan_result_id        关联本次提交的扫描结果
started_at / finished_at
```

### 14.4 scan_results(扫描结果)

```
id             主键
service        关联键
git_sha        关联键
scanner        sonarqube | semgrep | trivy
critical / high / medium / low   各级别问题数
passed         是否通过质量门禁
report_url     报告链接
created_at
```

### 14.5 configs + config_deliveries(配置版本与下发,设计细化)

> **存储取舍**:配置文件通常是文本(yaml/env/properties),几 KB~几十 KB,**默认落库(PostgreSQL)**——可与元数据同事务原子提交,diff 直接 SQL 取,备份/回滚统一。设阈值(如 1MB),**超过才溢出到对象存储**,库里存指针 + hash。回避原文档 `content` 字段"或指向对象存储"那个含糊的"或"。

**configs(配置版本,存全量快照)**

```
id             主键
service        关联键
env            环境
version        配置版本号(递增)
content        配置内容(文本直接入库;超阈值时存对象存储指针)
content_hash   sha256:用于去重、完整性校验、"下发的与库里是否一致"、幂等下发
diff_from      血缘指针(仅指向上一版本 id 用于展示 diff);content 存全量快照,
               任意版本可独立还原,不存增量 diff
operator       操作人
created_at
```

**config_deliveries(下发记录,一版本 × 多目标 × 结果)**

```
id                 主键
config_version_id  指向 configs
placement_id       下发到哪个放置点(见 service_placements)
status             pending | success | failed
result / error     下发结果
delivered_at
```

要点:
- **敏感配置严禁明文落库**(否则违背 §13"不落库明文"):`configs.content` 中密码/token 存 **占位符 + secret 引用**,真实值在凭证保险箱,下发时注入。
- **`content` 存全量快照,不存增量 diff**:`diff_from` 仅作血缘指针供展示计算 diff;存 diff 会让某版本无法独立还原。
- **下发状态从"单布尔 `applied`"升级为下发记录表**:一个配置版本会下发到多个实例,可能部分成功、部分失败,单标志表达不了这种多目标半成功态。

### 14.6 tasks(异步任务)

```
id             主键
type           deploy | rollback | start | stop | delete | update_config
target         service / server
payload        参数
status         pending | running | success | failed | unknown
               (unknown:命令超时/Agent 断连,可能已执行,待核对,见 §5.4)
result / error 执行结果
created_by     发起人
created_at / finished_at
```

### 14.7 audit_logs(审计)

```
id             主键
actor          操作人
action         动作(带资源与环境)
target         目标资源
env            环境
before / after 变更前后(如适用)
result         成功/失败
ip / ua        来源
created_at     不可篡改
```

### 14.8 alerts(告警)

```
id             主键
service        关联键(如可关联)
severity       critical | warning | info
summary        摘要
source         alertmanager
fired_at / resolved_at
status         firing | resolved
```

### 14.9 关系图(关联键落成外键)

```
scan_results.git_sha  ──┐
                        ├──▶ deployments (git_sha, scan_result_id)
services.name ──────────┤         │
                        │         ├──▶ tasks (target=service)
                        │         ├──▶ audit_logs
                        │         └──▶ (Prometheus 按 service label 关联指标)

```

## 15. 关键 API 契约

### 15.1 Webhook 接收(入向)

```
POST /api/webhooks/deployment    流水线部署完成上报(见 §8.2)
POST /api/webhooks/scan          扫描完成上报(见 §7.1)
```

### 15.2 服务操作(出向,写操作异步)

```

POST /api/services/{id}/stop         停止     → 返回 task_id
POST /api/services/{id}/restart      重启     → 返回 task_id
DELETE /api/services/{id}            删除     → 返回 task_id(高危,走审批)
POST /api/services/{id}/deploy       部署     → body: {version, strategy, env}
POST /api/services/{id}/rollback     回滚     → 读 previous_deployment_id
GET  /api/tasks/{task_id}            查任务进度
```

### 15.3 配置管理

```
GET  /api/services/{id}/configs                配置版本列表
GET  /api/services/{id}/configs/{version}      某版本内容
POST /api/services/{id}/configs                新增配置版本(暂存)
POST /api/services/{id}/configs/{v}/apply      下发生效 → task_id
POST /api/services/{id}/configs/rollback       回滚到上一版
```

### 15.4 查询与监控

```
GET  /api/servers                    服务器列表 + Agent 状态
GET  /api/services?env=prod          服务列表(按环境)
GET  /api/deployments?service=x      部署历史(主页 feed)
GET  /api/metrics/query              代理 PromQL 查询(资源指标)
```

### 15.5 Agent 通信协议(gRPC 双向流,示意)

```protobuf
service AgentService {
  // Agent → 控制面:建立长连,上报心跳与状态
  rpc Connect(stream AgentMessage) returns (stream ServerCommand);
}

message AgentMessage {
  oneof payload {
    Heartbeat heartbeat = 1;       // 心跳:agent 版本、机器在线
    ServiceStatus status = 2;      // 服务状态上报
    CommandResult result = 3;      // 命令执行结果回传
  }
}

message ServerCommand {
  string task_id = 1;
  oneof action {
    ExecCommand   exec = 2;        // 执行命令
    DeploySpec    deploy = 3;      // 部署
    ConfigUpdate  config = 4;      // 下发配置
  }
}
```

- 全链路关联(提交 → 扫描 → 部署 → 监控)
- 发布策略(金丝雀/蓝绿)+ 自动回滚
- 告警联动 + 通知

---

## 18. 端到端数据流

```
开发者 push 代码
   │
   ▼
代码扫描 ──webhook──▶ 控制面(存 scan_result, 挂到 git_sha)
   │
   ▼
流水线构建+部署 ──webhook──▶ 控制面(写 deployments 记录, source=pipeline)
   │                              │
   │                              ├─▶ 主页"最近部署" feed 实时刷新
   │                              ├─▶ Agent 心跳确认服务在新版本运行
   │                              └─▶ 在 Prometheus 曲线上打部署标记
   ▼
线上运行
   │
   ├── Agent 上报服务状态 ──▶ 主页"服务健康"
   ├── node_exporter → Prometheus ──▶ 主页"资源监控"
   └── 异常 → Alertmanager ──webhook──▶ 主页"告警" + 通知 + (可选)自动回滚
                                              │
用户在 UI 点回滚 ◀────────────────────────────┘
   │
   ▼
控制面读 previous_deployment_id → 重新部署上一版 artifact
(UI触发 → 又走一遍部署流程,形成闭环)
```

**一句话概括主线**:*一切改变线上的动作,不管从哪发起,最终都收敛成一条 deployment 记录;这条记录既能向前回溯到代码和扫描,又能向后关联到监控和告警,并统一显示在主页。*
